# 10~20대 아이돌 팬을 위한 UX/디자인 전략 — 기획 최종안

## 결론 요약

기존 K-F-C 코드베이스는 게임 메커니즘(클릭, 바통, 응원)이 견고하게 구축되어 있으나, **팬덤 감성 귀속**과 **외부 바이럴 장치**가 부재하여 10~20대 아이돌 팬의 정서적 몰입과 자발적 확산을 이끌어내지 못하고 있다. 전면 재설계 없이 **기존 WebSocket 이벤트와 Framer Motion 인프라 위에 감성 레이어를 쌓는 방식**으로, 팬이 "내 팀의 일원"임을 시각적으로 체감하고, 자랑할 수 있는 순간을 만들어주는 것이 핵심 전략이다.

---

## 채택한 아이디어

### 1. 콤보 배수 보너스 (창의팀 1순위 — 채택)

**채택 근거**: 현재 `game/page.tsx`에 combo 상태가 이미 존재하지만(L38, L172), 시각 피드백 전용이고 실제 점수에 반영되지 않는다. combo를 multiplier로 전환하면 게임 몰입도가 즉각 상승하며, 백엔드 변경이 최소화된다.

**구현 방향**:
- 프론트: combo 5 이상 시 click 이벤트에 `multiplier` 필드 추가 (1x/2x/3x)
- 백엔드: `game.gateway.ts` handleClick에서 multiplier 수신, `game.service.ts` addClick에서 INCRBY multiplier 적용
- 상한선: 최대 3x (어뷰징 방지 — 비평팀 지적 반영)

### 2. 팀 컬러 기반 감정 귀속 설계 (비평팀 필수사항 1번 + 창의팀 5순위 — 채택)

**채택 근거**: 양측 모두 동의한 핵심 사안. 현재 팀 컬러는 게이지 바와 TAP 버튼에만 적용되어 있고(`selectedSt!.colorCode`), 전체 UI가 팀 색으로 물드는 경험이 없다. 팬덤 서비스에서 "내 팀 색"은 정체성의 핵심이다.

**구현 방향**:
- 팀 선택 시 CSS 변수(`--team-primary`, `--team-secondary`)를 `<html>` 요소에 설정
- Confetti 컴포넌트의 하드코딩된 `COLORS` 배열(Confetti.tsx L5)을 팀 컬러 기반으로 교체
- 헤더, 카드 보더, 배경 그라데이션에 CSS 변수 적용

### 3. 일일 스코어 공유 카드 (창의팀 3순위 — 채택, 단계적)

**채택 근거**: 비평팀이 "반드시 해야 하는 것" 2번으로 SNS 공유 카드를 지목했고, 창의팀도 3순위로 제안. 외부 바이럴 없이는 폐쇄형 서비스에 머물게 된다. 다만 Canvas API 풀 구현은 2~3주 범위이므로 단계적 접근.

**구현 방향**:
- 1단계 (1주): og:image 동적 생성 — Next.js App Router의 `opengraph-image.tsx` 활용
- 2단계 (2주): Web Share API로 모바일 네이티브 공유
- 3단계 (3주): Canvas 기반 개인 기여 카드 (내 클릭수, 팀 순위, 달성률)

### 4. 실시간 참여자 카운터 (창의팀 4순위 — 채택)

**채택 근거**: 구현 난이도 하, 경쟁 심리 자극 효과 높음. `score:update` 이벤트에 `onlineCount` 필드를 추가하면 된다.

**구현 방향**:
- 백엔드: `game.gateway.ts`의 `handleConnection`/`disconnect`에서 Redis INCR/DECR로 접속자 수 관리
- `score:update` 이벤트 payload에 `onlineCount` 포함
- 프론트: 게임 헤더 영역에 "지금 N명 참여 중" 뱃지 표시

### 5. 팀 컬러 탄막 오버레이 (창의팀 2순위 — 조건부 채택)

**채택 근거**: 팬덤 감성의 핵심 레이어이나, 모바일 성능 이슈(비평팀 #7)를 고려하여 조건부 채택. `cheer:new` 이벤트를 재활용하되 성능 가드레일을 설정한다.

**구현 방향**:
- 응원 게시판(`cheer/page.tsx`)의 `cheer:new` 이벤트를 게임 페이지에서도 구독
- 탄막은 최대 동시 15개, overflow 시 FIFO 제거
- 저사양 기기 감지(navigator.hardwareConcurrency <= 4) 시 자동 비활성화
- 사용자 토글 옵션 제공

## 반영한 리스크 대응

### 어뷰징 방지 (비평팀 필수사항 3번 — 최우선 반영)

**현황 분석**: 코드베이스 검토 결과, 응원 게시판에만 IP 기반 rate limit이 있고(`cheer.service.ts` L11: 5초 쿨다운), **클릭 게임에는 rate limit이 전혀 없다**. `addClick`은 무한 INCR이 가능한 상태다.

**대응 방안**:
- **즉시**: Redis 기반 IP별 초당 클릭 상한 설정 (20 clicks/sec) — `game.service.ts`에 sliding window rate limiter 추가
- **1주 내**: fingerprint.js 연동하여 브라우저 핑거프린트 기반 이중 검증
- **2주 내**: 비정상 패턴 감지 (일정 간격 정확한 클릭 = 매크로) — Redis에 클릭 간격 기록 후 표준편차 기반 필터링

### SaaS/B2B 톤 제거 (비평팀 즉시수정 #1)

**현황**: 페이지 타이틀이 "아이돌 팬덤 대회"로 무미건조. Admin 관련 용어가 일부 노출.

**대응**:
- 카피라이팅 전면 교체: "아이돌 팬덤 대회" → 시즌별 테마 타이틀 (예: "제1회 아이돌 가을 운동회")
- "활성 시즌이 없습니다" 등 관리자 용어 → "다음 운동회를 준비 중이에요!" 등 팬 친화적 표현
- `layout.tsx`의 metadata 전면 수정

### Hero 영역 활성화 (비평팀 즉시수정 #4)

**대응**: 메인 페이지(`page.tsx`)에 시즌 카운트다운 타이머 + 현재 1위 팀 실시간 표시 + "참여하기" CTA 버튼 추가

### 모바일 UX 최적화 (비평팀 전략필요 #7)

**대응**:
- TAP 버튼 영역 확대 (현재 w-48/h-48 → 모바일에서 w-56/h-56)
- 하단 네비게이션 바 터치 영역 최소 44px 보장
- 게이지 바 수치 가독성 개선 (현재 text-xs → text-sm)
- viewport에 이미 `userScalable: false` 설정됨 (확인 완료)

## 하지 않을 것

### 1. 배지/미션/스트릭 시스템 도입 금지

**근거**: 비평팀이 명확히 지적한 "하면 안 되는 것" 1번. 과금 게임 느낌을 주며, 7일 시즌제 특성상 스트릭 압박은 팬 이탈을 유발한다. 콤보 배수 보너스만으로 단기 몰입 메커니즘은 충분하다.

### 2. 커뮤니티 기능 MVP 포함 금지

**근거**: 빈 게시판은 서비스 미완성 인상을 준다. 응원 게시판이 이미 실시간 소통 역할을 하고 있으므로, 별도 커뮤니티는 MAU 1만 이상 달성 후 검토한다.

### 3. 아바타/캐릭터 시스템

**근거**: 창의팀이 "신규 기능"으로 분류한 항목. 디자인 리소스가 과다하고, 아이돌 팬 서비스의 핵심은 아이돌 자체이지 아바타가 아니다. IP 라이선스 없이 캐릭터를 만들면 오히려 저렴한 인상을 준다.

### 4. 스포츠/에너지 비주얼 과도한 적용

**근거**: 비평팀 "하면 안 되는 것" 3번. 10~20대 아이돌 팬 감성은 스포츠 피트니스가 아니라 **아이돌 콘서트/팬미팅** 분위기에 가깝다. 운동회 컨셉은 유지하되, 비주얼 톤은 축제/파티 방향으로 설정한다.

## 리스크 관리

### 1. 어뷰징 (심각도: 상)

- **위험**: 클릭 게임 특성상 매크로/다중 탭 어뷰징 필연적
- **현황**: 클릭에 rate limit 없음 (치명적)
- **대응**: Phase 1-1에서 즉시 rate limiter 투입. IP 해시 + fingerprint 이중 검증. 비정상 패턴 감지는 Phase 2-5에서 고도화
- **실패 사례 참고**: 뮤빗 부정 투표 사태 — 신뢰 붕괴 시 복구 불가

### 2. 모바일 성능 (심각도: 중)

- **위험**: 탄막 + Confetti + Framer Motion 동시 작동 시 저사양 기기 프레임 드롭
- **대응**: 탄막 동시 15개 상한, hardwareConcurrency 기반 자동 비활성화, 사용자 토글
- **모니터링**: Performance Observer API로 FPS 수집

### 3. 재방문 이탈 (심각도: 중)

- **위험**: 7일 시즌제이므로 중반 이후 관심 하락
- **대응**: 시즌 카운트다운 + 순위 변동 알림(rank:change 이벤트) + 일일 공유 카드로 재유입
- **실패 사례 참고**: BTS World — IP만으로는 리텐션 불가, 게임성이 뒷받침되어야 함

### 4. 기술 제약 (심각도: 하)

- **위험**: Redis 단일 인스턴스 기반이므로 동시 접속 급증 시 병목
- **대응**: 현재 2초 간격 DB flush는 적절. 동시 접속 500명 초과 시 Redis Cluster 전환 검토
- **선행 조건**: Phase 1에서 실시간 접속자 카운터 구현하여 트래픽 패턴 파악

## 보류/추가 검토 사항

1. **PWA 도입 시점**: 재방문율 데이터 확보 후 판단 (Phase 2 이후)
2. **개인 랭킹/통계**: DB 스키마 변경(ipHash별 집계 테이블) 필요 — 백엔드팀 공수 확인 후 Phase 3에서 결정
3. **운동회 포스터 배경/테마**: 디자인팀 리소스와 시즌 컨셉 확정 후 결정
4. **알림 시스템**: 푸시 알림은 PWA 도입과 연계. 당분간 인앱 배너 + 시즌 카운트다운으로 대체
5. **유료화 요소**: 비평팀의 Weverse 사례 경고를 수용 — MVP 단계에서 유료화 논의 금지. 광고(AdSense 이미 연동됨)로 수익 모델 검증 선행
